+    197:

+    199: 
+    200: **Benchmark estimates for `npm install` (medium project, ~500 packages)**:
+    201: 
+    202: | Throughput Mode | 1 user | 10 concurrent | 30 concurrent |

+    213: 4. 30 users × 1,000 write IOPS each = 30,000 IOPS → hits the 7,000 write IOPS limit

+    230: ### 3.4 EFS Access Points for Multi-Tenant Isolation
+    231: 
+    232: **Current state**: The code uses directory-based isolation (`/users/{subdomain}/`) via `entrypoint.sh`. The EFS task definition mounts `rootDirectory: '/'` and the entrypoint creates per-user directories.
+    233: 
+    234: **Problem**: Any container can read/write ANY user's directory. The `chown` in entrypoint.sh is cosmetic — all containers run as the same UID.
+    235: 
+    236: **Severity**: 🔴 HIGH — Cross-tenant data access possible
+    237: 
+    238: **Recommendation**: Use **EFS Access Points** for proper isolation:
+    239: 
+    240: ```typescript
+    241: // Per-user Access Point (create during user provisioning)
+    242: const accessPoint = new efs.AccessPoint(this, `AP-${subdomain}`, {
+    243:   fileSystem,
+    244:   path: `/users/${subdomain}`,
+    245:   createAcl: { ownerUid: '1000', ownerGid: '1000', permissions: '750' },
+    246:   posixUser: { uid: '1000', gid: '1000' },
+    247: });
+    248: ```
+    249: 
+    250: Then in the task definition's EFS volume config:
+    251: ```typescript
+    252: efsVolumeConfiguration: {
+    253:   fileSystemId: fileSystem.fileSystemId,
+    254:   transitEncryption: 'ENABLED',
+    255:   authorizationConfig: {
+    256:     accessPointId: accessPoint.accessPointId,
+    257:     iam: 'ENABLED',
+    258:   },
+    259: }
+    260: ```
+    261: 
+    262: **Benefits**:
+    263: - Kernel-level enforcement — container literally cannot see other users' files
+    264: - Automatic directory creation with correct permissions
+    265: - IAM authorization adds another layer
+    266: - No code changes needed in entrypoint.sh
+    267:
