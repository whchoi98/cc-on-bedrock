## 1. Hardcoded Dependencies

### [Critical] Hardcoded AWS Account ID `061525506239`

Production account ID is embedded in multiple source files. Anyone cloning this repo gets a non-functional or dangerous deployment pointing at the wrong account.

| File | Line | Value |
|------|------|-------|
| `cdk/cdk.context.json` | 2 | `availability-zones:account=061525506239` |
| `cdk/lib/05-dashboard-stack.ts` | 122 | `s3://cc-on-bedrock-deploy-061525506239/dashboard/...` |
| `shared/nextjs-app/src/lib/aws-clients.ts` | 52 | `const accountId = process.env.AWS_ACCOUNT_ID ?? "061525506239"` |
| `shared/nextjs-app/src/app/api/ai/runtime/route.ts` | 16 | `arn:aws:bedrock-agentcore:ap-northeast-2:061525506239:runtime/...` |
| `agent/lambda/create_targets.py` | 18 | `ACCOUNT_ID = os.environ.get("ACCOUNT_ID", "061525506239")` |
| `agent/CLAUDE.md` | 30-41 | Multiple ECR URIs and ARNs with account ID |

**Fix**: Replace all hardcoded account IDs with environment variables or CDK/Terraform intrinsics (`cdk.Aws.ACCOUNT_ID`, `data.aws_caller_identity`). The S3 bucket in UserData should use `aws sts get-caller-identity` at runtime.

### [High] Hardcoded Domain `whchoi.net`

A real domain is committed as the CDK default config and in test scripts.

| File | Line | Value |
|------|------|-------|
| `cdk/config/default.ts` | 34 | `domainName: 'whchoi.net'` |
| `scripts/create-test-users-30.sh` | 8, 33 | `cconbedrock-dashboard.whchoi.net`, `@whchoi.net` |
| `shared/nextjs-app/src/app/security/security-dashboard.tsx` | 429 | `ACM *.whchoi.net` |
| `docs/architecture.md` | 13-160 | Multiple references |

**Fix**: Change CDK default to `example.com` (matching Terraform/CFN). Move real domain to `cdk.context.json` or environment variable. Parameterize test scripts.

### [High] Hardcoded CloudFront Prefix List ID `pl-22a6434b`

This prefix list ID is specific to `ap-northeast-2`. Deploying to any other region will fail silently (wrong prefix list) or break security group rules.

| File | Occurrences |
|------|-------------|
| `cdk/lib/04-ecs-devenv-stack.ts` | Line 261 |
| `cdk/lib/05-dashboard-stack.ts` | Line 78 |
| `terraform/modules/ecs-devenv/main.tf` | Line 368 |
| `terraform/modules/dashboard/main.tf` | Line 20 |
| `cloudformation/params/default.json` | Line 16 |

**Fix**: Look up the prefix list dynamically. In Terraform: `data "aws_ec2_managed_prefix_list" "cloudfront" { name = "com.amazonaws.global.cloudfront.origin-facing" }`. In CDK: use `ec2.Peer.prefixList()` with a config parameter. CloudFormation already parameterizes it but the default is still hardcoded.

## 2. Security Issues

### [Critical] Hardcoded Default Password in Source Code

`shared/nextjs-app/src/lib/aws-clients.ts` line 340:
```typescript
{ name: "CODESERVER_PASSWORD", value: process.env.CODESERVER_PASSWORD ?? "CcOnBedrock2026!" },
```

A default password is embedded in application source code. If the `CODESERVER_PASSWORD` env var is not set, every user container gets the same known password.

**Fix**: Generate a per-user random password at container start time (e.g., from Secrets Manager or `openssl rand`). Never commit default passwords.

---

### [High] Wildcard Bedrock IAM Permissions (`Resource: *`)

Multiple roles grant `bedrock:InvokeModel` and `bedrock:InvokeModelWithResponseStream` on `Resource: *`, allowing invocation of any model in any region — not just the intended Claude models.

| File | Role | Line(s) |
|------|------|---------|
| `cdk/lib/02-security-stack.ts` | LiteLLM EC2 Role | 120 |
| `cdk/lib/04-ecs-devenv-stack.ts` | ECS Instance Role | 147 |
| `cdk/lib/05-dashboard-stack.ts` | Dashboard EC2 Role | 41 |
| `cloudformation/02-security.yaml` | LitellmEc2Role | 197 |
| `cloudformation/04-ecs-devenv.yaml` | EcsTaskRole | 66 |
| `terraform/modules/security/main.tf` | litellm_ec2, dashboard_ec2 | 235, 323 |
| `terraform/modules/ecs-devenv/main.tf` | ecs_task | 24 |

Note: The CDK ECS Task Role (04-ecs-devenv-stack.ts lines 50-65) correctly scopes to `anthropic.claude-*` patterns — this is the right approach. But the ECS Instance Role, LiteLLM role, and Dashboard role do not.

**Fix**: Scope all Bedrock permissions to specific model ARN patterns:
```
arn:aws:bedrock:*::foundation-model/anthropic.claude-*
arn:aws:bedrock:*:<ACCOUNT>:inference-profile/*anthropic.claude-*
```

---

### [High] Wildcard ECS/ELB Permissions on Dashboard Role

`cdk/lib/02-security-stack.ts` lines 162-173:
```typescript
actions: ['ecs:RunTask', 'ecs:StopTask', 'ecs:DescribeTasks', 'ecs:ListTasks', 'ecs:TagResource'],
resources: ['*'],
// ...
actions: ['elasticloadbalancing:CreateTargetGroup', ...],
resources: ['*'],
```

The Dashboard EC2 role can manage any ECS task or ALB target group in the account, not just the `cc-on-bedrock-devenv` cluster.

**Fix**: Scope ECS actions to the specific cluster ARN. Scope ELB actions to resources tagged with `Project=cc-on-bedrock` or use resource ARN patterns.

---

### [High] EFS Shared Root — Cross-User File Access Risk

`cdk/lib/04-ecs-devenv-stack.ts` lines 195-200:
```typescript
efsVolumeConfiguration: {
  fileSystemId: fileSystem.fileSystemId,
  rootDirectory: '/',  // Override per-user at RunTask: /users/{subdomain}
  transitEncryption: 'ENABLED',
},
```

The task definition mounts the entire EFS root at `/home/coder`. The comment says "override per-user at RunTask" but the entrypoint script only creates subdirectories — it doesn't prevent a user from navigating to `/home/coder/users/other-user/`. All 3 IaC implementations have this issue.

**Fix**: Use EFS Access Points with per-user `rootDirectory` and `posixUser` enforcement. Create access points dynamically when starting containers, and reference them in the EFS volume configuration. This provides kernel-level isolation.

---

### [High] `unsafeUnwrap()` Exposes Secret in CloudFormation Template

`cdk/lib/05-dashboard-stack.ts` line 196:
```typescript
'X-Custom-Secret': cloudfrontSecret.secretValue.unsafeUnwrap(),
```

`unsafeUnwrap()` resolves the secret at synth time and embeds the plaintext value in the CloudFormation template. Anyone with `cloudformation:GetTemplate` permission can read it.

**Fix**: Use `{{resolve:secretsmanager:...}}` dynamic references (as CloudFormation templates already do correctly), or pass the secret ARN and resolve it at runtime.

---

### [High] CloudFront Distributions Missing Custom Domain + WAF

Both DevEnv and Dashboard CloudFront distributions (across all 3 IaC tools) use `cloudfront_default_certificate = true` / `CloudFrontDefaultCertificate: true` without custom domain aliases or WAF.

Issues:
- No WAF protection against common web attacks (SQLi, XSS, bot traffic)
- Default `*.cloudfront.net` domain means no custom SSL certificate validation
- No geo-restriction configured (may be intentional for global access)

**Fix**: Attach ACM certificates (us-east-1) with domain aliases. Associate an AWS WAF WebACL with rate limiting and managed rule groups.

---

### [Medium] ALB HTTP Listener Open Without CloudFront Validation

`cdk/lib/04-ecs-devenv-stack.ts` lines 275-281:
```typescript
this.alb.addListener('HttpListener', {
  port: 80,
  protocol: elbv2.ApplicationProtocol.HTTP,
  defaultAction: elbv2.ListenerAction.fixedResponse(403, ...),
});
```

The DevEnv ALB has an HTTP listener on port 80 but the security group only allows CloudFront prefix list on port 443. The HTTP listener doesn't validate the `X-Custom-Secret` header, so if someone bypasses CloudFront and reaches the ALB directly on port 80, they get a 403 — which is fine. However, the ALB SG also has:
```typescript
albSg.addIngressRule(ec2.Peer.anyIpv4(), ec2.Port.tcp(80), 'Allow HTTP (CloudFront http-only origin)');
```
This opens port 80 to the entire internet, bypassing the CloudFront prefix list restriction.

**Fix**: Remove the `0.0.0.0/0` port 80 ingress rule. If CloudFront needs HTTP origin, restrict port 80 to the CloudFront prefix list as well.

---

### [Medium] Dashboard UserData Embeds Cognito IDs and VPC ID

`cdk/lib/05-dashboard-stack.ts` lines 115-140 write Cognito Client ID, User Pool ID, and VPC ID directly into the `.env` file via UserData. While these aren't secrets per se, they're visible in the EC2 console and CloudFormation template.

**Fix**: Fetch these from SSM Parameter Store or Secrets Manager at boot time, similar to how `NEXTAUTH_SECRET` is already fetched.

---

### [Medium] Containers Run as Root by Default

Both `Dockerfile.ubuntu` and `Dockerfile.al2023` don't specify a `USER` directive. The entrypoint script runs as root and then uses `sudo -u coder` for code-server. But the container process itself starts as root, and any escape from code-server runs as root.

**Fix**: Add `USER coder` at the end of Dockerfiles. Restructure entrypoint to handle root-required setup in an init container or use `gosu` for the final exec.

---

### [Medium] DLP "Restricted" Policy Allows All HTTPS Outbound

`cdk/lib/04-ecs-devenv-stack.ts` lines 108-109:
```typescript
sgRestricted.addEgressRule(ec2.Peer.anyIpv4(), ec2.Port.tcp(443), 'Allow HTTPS for whitelisted domains');
```

The "restricted" DLP policy allows HTTPS to `0.0.0.0/0`. The comment says "whitelisted domains" but security groups can't filter by domain — only by IP/CIDR. This effectively allows all HTTPS traffic, making "restricted" nearly identical to "open" for HTTPS.

**Fix**: Use Route 53 DNS Firewall or AWS Network Firewall with domain-based rules for actual domain whitelisting. The security group alone cannot enforce this.

---

### [Medium] No HTTPS Between ALB and ECS Containers

All task definitions expose port 8080 (HTTP) and ALBs forward to port 8080 over HTTP. Traffic between ALB and containers is unencrypted within the VPC.

**Fix**: For sensitive development environments, consider TLS termination at the container level or use AWS App Mesh with mTLS.

---

### [Medium] Cognito Password Policy Missing Symbol Requirement

All 3 IaC implementations set `requireSymbols: false`. For a platform hosting development environments with access to Bedrock AI models, this weakens password strength.

**Fix**: Enable `requireSymbols: true` and increase minimum length to 12.

---

### [Medium] RDS Single-AZ Deployment

`cdk/lib/03-litellm-stack.ts` line 97: `multiAz: false`. A single-AZ RDS instance means database downtime during AZ failures or maintenance windows.

**Fix**: Enable Multi-AZ for production deployments. Keep single-AZ as an option for dev/education environments via config parameter.

---

### [Low] ECR Image Tag Mutability

`terraform/modules/ecs-devenv/main.tf` line 72: `image_tag_mutability = "MUTABLE"`. This allows overwriting existing image tags, which could be exploited for supply chain attacks.

**Fix**: Set `image_tag_mutability = "IMMUTABLE"` and use unique tags (git SHA) instead of `latest`.

---

### [Low] Missing `readonlyRootFilesystem` on Containers

None of the ECS task definitions set `readonlyRootFilesystem: true`. This allows containers to write to the root filesystem, increasing the attack surface.

**Fix**: Enable read-only root filesystem and use tmpfs mounts for writable paths (`/tmp`, `/var/run`).

---

### [Low] Terraform State Not Configured for Remote Backend

`terraform/providers.tf` has no `backend` block. State defaults to local file, which is unsuitable for team use and risks state loss.

**Fix**: Add S3 + DynamoDB backend configuration.

---

## 3. Sensitive Data Exposure

### [Critical] Hardcoded Account ID in Committed `cdk.context.json`

`cdk/cdk.context.json` line 2:
```json
"availability-zones:account=061525506239:region=ap-northeast-2": [...]
```

This file is committed to git and exposes the AWS account ID. While account IDs aren't secrets, they enable targeted attacks (cross-account role assumption attempts, resource enumeration).

**Fix**: Add `cdk.context.json` to `.gitignore` (CDK regenerates it on `cdk synth`). Or strip account-specific entries before committing.

---

### [High] `.gitignore` Pattern `*.env` May Miss `.env.local`

The gitignore uses `*.env` which matches `something.env` but does NOT match `.env` (dotfiles without extension) or `.env.local`, `.env.production`.

**Fix**: Add explicit entries:
```
.env
.env.*
!.env.example
```

---

### [Medium] Terraform `random_password` Values in State

`terraform/modules/security/main.tf` uses `random_password` resources for secrets. These plaintext values are stored in Terraform state.

**Fix**: Ensure Terraform state is encrypted (S3 backend with SSE-KMS). Consider using `aws_secretsmanager_secret` with `generate_secret_string` instead of `random_password` to keep secrets out of state entirely.

---

### [Low] Documentation Files Contain Real Infrastructure Details

Files in `output/` and `docs/` contain real account IDs, domain names, and ARNs. These are informational but increase the attack surface if the repo is public.

**Fix**: Sanitize documentation files or move them to a private wiki. Add `output/` to `.gitignore`.

---

## Summary

| Severity | Count | Key Areas |
|----------|-------|-----------|
| Critical | 3 | Hardcoded account ID, hardcoded default password, account ID in cdk.context.json |
| High | 8 | Domain exposure, prefix list ID, wildcard IAM, EFS isolation, unsafeUnwrap, CloudFront config, ALB HTTP open, gitignore gaps |
| Medium | 8 | Hardcoded ARNs/regions, DLP weakness, no container HTTPS, password policy, RDS single-AZ, root containers, UserData secrets, TF state |
| Low | 5 | Cognito domain prefix, ECR mutability, readonly rootfs, TF backend, docs exposure |

### Top 3 Priorities

1. **Remove hardcoded account ID and password** from source code — these are the most immediately exploitable issues
2. **Implement EFS Access Points** for per-user isolation — current setup allows cross-user file access
